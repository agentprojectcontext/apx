Un **runtime** es un CLI de IA externo que APX puede invocar en modo headless en nombre de un
agente. Usás `apx run` para delegarle un prompt a una de estas herramientas: APX construye el
system prompt, lanza el CLI, captura su salida y guarda un registro de sesión. La herramienta
externa se encarga de la interacción real con el modelo, las ediciones de archivos y los comandos
de shell — APX registra el envoltorio.

Esto es distinto de `apx exec`, que llama a un motor LLM directamente dentro de APX sin lanzar
ningún proceso externo. Para ese camino, consultá [Engines](/apx/es/engine/engines/).

## Runtimes soportados

| ID del runtime | Binario | Qué lanza APX |
|----------------|---------|--------------|
| `claude-code` | `claude` | Modo headless `-p`; recibe el system prompt del agente via `--append-system-prompt`; devuelve JSON estructurado con un `session_id` que apunta al transcript de Claude Code |
| `codex` | `codex` | `codex exec --sandbox workspace-write --skip-git-repo-check`; el system prompt se antepone al prompt del usuario (no hay flag `--system` en modo exec) |
| `opencode` | `opencode` | `opencode run`; system prompt antepuesto al cuerpo del prompt |
| `aider` | `aider` | `aider --message --yes-always --no-auto-commits`; no interactivo, sin commits automáticos |
| `cursor-agent` | `cursor-agent` | `--print --output-format text --trust --force`; modo print headless |
| `gemini-cli` | `gemini` | `--prompt --output-format text --approval-mode yolo`; modo prompt headless |
| `qwen-code` | `qwen` | `--output-format text --approval-mode yolo`; system prompt via `--append-system-prompt` |

<Aside type="note">
  APX pasa el system prompt del agente y el prompt del usuario. **No** elige ni sobreescribe el
  modelo dentro del CLI externo — la configuración propia del CLI (ej. los defaults de `CLAUDE.md`
  para Claude Code, o el archivo de config de Codex) es la que manda.
</Aside>

## Verificar qué está instalado

Antes de elegir un runtime, confirmá cuáles CLIs están disponibles en tu máquina:

```bash
apx env detect
```

<Screenshot

  hint="Run: apx env detect" />

La salida agrupa los resultados por categoría: `runtime`, `engine` y `tool`. Un runtime que no
aparece en la lista no está instalado — instalá su binario primero y luego ejecutá `apx env detect`
de nuevo.

## Ejecutar un agente

```bash
apx run <agent> --runtime <id> "<prompt>" [--timeout <seconds>] [--project <name|id|path>]
```

**Ejemplos:**

```bash
# Code review completo via Claude Code
apx run reviewer --runtime claude-code "Review the diff in src/ for memory leaks"

# Tarea de refactoring via Codex con timeout extendido
apx run cody --runtime codex "Refactor parseAgentsMd to use a state machine" --timeout 600

# Una sola línea con OpenCode
apx run sofia --runtime opencode "Add JSDoc to every exported function in lib/"
```

### Opciones

| Flag | Descripción |
|------|-------------|
| `--runtime <id>` | Requerido. Uno de los IDs de runtime de la tabla de arriba |
| `--timeout <seconds>` | Tiempo máximo de pared antes de que APX mate el subproceso |
| `--project <name\|id\|path>` | Asociar a un proyecto específico; por defecto usa el directorio actual |

## Cómo APX inyecta el system prompt

Cuando ejecutás `apx run`, APX hace lo siguiente:

<Steps>

1. Lee la definición del agente desde `.apc/agents/<slug>.md` junto con su `memory.md` y los
   skills cargados para construir el system prompt completo.

2. Llama a `buildAgentSystem({ invocation: "runtime", runtime: "<id>" })` — el mismo builder que
   usa `apx exec` y `apx chat`, pero con el tag de invocación `runtime` para que los prompts
   puedan ramificarse.

3. Pasa el resultado al mecanismo de inyección del CLI (ej. `--append-system-prompt` para
   `claude-code` y `qwen-code`, o antepuesto al cuerpo del prompt para los runtimes que no tienen
   un flag dedicado).

4. Captura el stdout. Si la salida incluye una línea que empieza con `APC_RESULT:`, ese valor se
   guarda como campo de resultado estructurado de la sesión.

5. Escribe un archivo de sesión en
   `~/.apx/projects/<apx-id>/agents/<slug>/sessions/<date>-<runtime>-<id>.md` con un frontmatter
   que apunta al transcript propio de la herramienta externa (cuando el runtime lo provee).

</Steps>

## Registros de sesión y cómo retomar

Después de que `apx run` termina, APX crea un registro de sesión:

```yaml
# ~/.apx/projects/<apx-id>/agents/reviewer/sessions/2026-05-27-claude-code-abc123.md
---
runtime: claude-code
session_id: abc123
external_session_path: /Users/me/.claude/projects/<encoded-cwd>/abc123.jsonl
---
```

El campo `external_session_path` apunta al transcript propio de la herramienta externa. Podés
leerlo, resumirlo o continuar desde él con los comandos de sesión:

```bash
# Listar sesiones recientes de runtime para un agente
apx session list reviewer

# Resumir qué pasó en una sesión
apx session resume <id> --summary

# Leer el transcript completo
apx session get <id> --full

# Continuar la sesión en el CLI nativo de Claude Code
apx session resume <id> --continue
```

## Resultados estructurados con APC_RESULT

Si querés que APX capture un valor legible por máquina desde el runtime externo, instruí al modelo
(via el prompt) para que imprima en su última línea:

```text
APC_RESULT: <valor en una línea>
```

APX parsea esto y lo guarda como campo `result` de la sesión, que otras rutinas o el super-agente
pueden leer.

## run vs. exec

| | `apx run` | `apx exec` |
|---|---|---|
| **Qué ejecuta** | CLI externo (claude, codex, …) | Llamada directa al LLM dentro de APX |
| **Ediciones / shell** | Sí — el CLI externo puede escribir archivos y ejecutar comandos | No — respuesta de LLM de solo lectura |
| **Elección de modelo** | Controlado por la config propia del CLI externo | Controlado por `--model` o el modelo configurado del agente |
| **Transcript de sesión** | Guardado en APX y en la ruta propia de la herramienta externa | Guardado solo en conversaciones de APX |
| **Usalo cuando** | Necesitás capacidades de coding-agent completas (ediciones, terminal) | Necesitás una respuesta rápida o generación puntual |

<Aside type="caution">
  `apx run` es siempre **headless** — el CLI externo corre de forma no interactiva. Para sesiones
  de codificación interactivas, invocá el CLI directamente (ej. `claude` para Claude Code). Evitá
  también prompts muy grandes por línea de comandos; los límites de argumentos de shell aplican.
</Aside>
