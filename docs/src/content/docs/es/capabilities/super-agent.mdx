El **super-agent** es un _modo_, no un nombre de persona. Cuando ejecutás `apx exec` sin apuntar
a un agente de proyecto específico, APX activa este modo: el loop de herramientas a nivel daemon
que puede alcanzar todas las herramientas registradas — operaciones de archivos, llamadas a MCP,
Telegram, shell, tareas, sesiones, y más.

El nombre que el asistente usa al hablar con vos viene de `~/.apx/identity.json` (default
**"APX"**). Las referencias a "super-agent" son un término técnico interno para el modo. Lo
configurás con las claves `super_agent.*` en `~/.apx/config.json`, y los kinds de routines
pueden seguir diciendo `super_agent` — pero los usuarios nunca ven esa etiqueta.

<Aside type="note">
  Este diseño está documentado en `spec/decisions/002-super-agent-is-mode-not-name.md`.
  Ver [Configuración](/apx/es/concepts/configuration/) para la referencia completa de `super_agent.*`.
</Aside>

## Habilitarlo

El modo está deshabilitado por defecto. Seteá estas dos claves antes de usarlo por primera vez:

```bash
apx config set super_agent.enabled true
apx config set super_agent.model "anthropic:claude-sonnet-4-5"
```

Una vez habilitado, cada llamado a `apx exec` sin `-a <agent>` pasa por este modo.

## `apx exec` — el punto de entrada de una sola pasada

```bash
apx exec "<prompt>"                         # modo super-agent (default)
apx exec -- "<prompt>"                      # igual; -- evita problemas de parseo de flags
apx exec -a reviewer "<prompt>"            # agente de proyecto nombrado en su lugar
apx exec "<prompt>" --model gpt-5          # sobrescribir el modelo configurado para esta llamada
apx exec "<prompt>" --max-tokens 2000      # limitar tokens de salida
apx exec "<prompt>" --project my-app       # anclar al contexto de un proyecto específico
```

Cuando el modo super-agent está activo, el daemon resuelve el nombre del asistente desde
`~/.apx/identity.json` y lo muestra en las líneas de estado, encabezados del TUI y respuestas de
Telegram — nunca la cadena "super-agent".

## El loop de herramientas

Por debajo, `runSuperAgent()` construye un system prompt (bloque de identidad + inventario de
proyectos + bloque de memoria + bloque de hilos activos), selecciona un subconjunto de schemas de
herramientas apropiado para el canal, y llama al motor via el loop estándar `runAgent()`. El loop
continúa hasta que el modelo deja de llamar herramientas o una señal lo aborta.

### Registro de herramientas

El registro completo tiene unos 30 herramientas nativas más herramientas con bridge del registro
(browser, fetch, search, glob, grep). El daemon elige un subconjunto según el canal para mantenerse
dentro de los límites de TPM de los tiers económicos:

| Canal | Conjunto de herramientas | Motivo |
|---|---|---|
| `api` / `apx exec` | Completo | Deliberado, modelo elegido por el usuario |
| `routine` | Completo | Schedulado, autónomo, modelo elegido por el usuario |
| `web` / `code` | Completo | Workspace de larga duración |
| `telegram` | Core (~700 tokens) | Tiers económicos, respuestas rápidas |
| `desktop` / `deck` | Core | Igual |

El modelo siempre puede expandir su superficie llamando a `load_skill`.

### Herramientas nativas seleccionadas

| Herramienta | Qué hace |
|---|---|
| `create_task` / `list_tasks` | Agregar y leer items TODO por proyecto |
| `call_agent` | Delegar a un agente de proyecto nombrado sin spawnear un nuevo proceso |
| `call_mcp` | Llamar cualquier herramienta de servidor MCP registrado |
| `call_runtime` | Ejecutar una sesión de Claude Code / Codex |
| `run_shell` | Ejecutar un comando shell (con gate de permission mode) |
| `send_telegram` | Enviar un mensaje via el plugin de Telegram |
| `remember` | Escribir un hecho durable en `~/.apx/memory.md` |
| `search_sessions` | Recuperar transcriptos de sesiones pasadas |
| `set_identity` | Actualizar campos de `~/.apx/identity.json` |

## Brokering cross-channel

El modo super-agent es el handler común detrás de cada superficie: mensajes de Telegram, prompts
de la ventana Desktop, llamados a `apx exec`, chat web y ejecuciones de routines todos confluyen
en `runSuperAgent()`. Cada superficie pasa un identificador `channel` (`telegram`, `desktop`,
`web`, `routine`, `api`, …) que determina:

- Qué subconjunto de herramientas es visible para el modelo.
- Si se inyecta un bloque de "hilos activos" (turnos recientes en _otros_ canales).
- Qué reglas de identidad del sender y de role-gating de Telegram aplican.

El memory broker corre antes de cada llamada no tool-free e inyecta un bloque `[MEMORIA
RELEVANTE]` en el system prompt. Ver [Sistema de memoria](/apx/es/capabilities/memory-system/)
para los detalles.

## Identidad — `apx identity`

El nombre visible del asistente se almacena en `~/.apx/identity.json`:

```json
{
  "agent_name": "APX",
  "owner_name": ""
}
```

Tres comandos lo gestionan:

```bash
apx identity show                 # imprimir los campos de identidad actuales
apx identity set agent_name Ada   # cambiar el nombre visible a "Ada"
apx identity set owner_name Sam   # setear el nombre del owner/usuario
apx identity wizard               # asistente de setup interactivo
```

## Permission mode

`permission_mode` en `super_agent` controla qué puede hacer el loop de herramientas sin preguntar:

| Modo | Comportamiento |
|---|---|
| `total` | Todas las herramientas corren sin confirmación |
| `automatico` | APX decide automáticamente (recomendado) |
| `permiso` | Solo las herramientas en `allowed_tools`; todo lo demás pregunta |

```bash
apx permission set automatico
```

## Fallback de modelo

Si el modelo primario no está disponible, APX recorre una lista ordenada configurable de providers
de fallback. Seteá el orden y habilitá el router:

```bash
apx model enable
apx model order ollama openrouter groq
apx model status          # probá todos los providers; mostrá cuál se usaría ahora
```

Ver [Configuración](/apx/es/concepts/configuration/) para el schema completo de
`super_agent.model_fallback`.
