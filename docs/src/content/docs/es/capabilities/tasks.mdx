APX tiene una **lista TODO por proyecto** respaldada por un event log append-only. Las tasks están
asociadas a un proyecto, son direccionables por prefijo de short-id y nunca se borran realmente
— las transiciones de estado (`done`, `drop`) se registran como eventos y la task persiste para
siempre.

## Almacenamiento

Las tasks viven en `~/.apx/projects/<apxId>/tasks/YYYY-MM.jsonl`, un archivo por mes. El estado
es el fold del event stream: crear, completar, dropear, reabrir y patchear agregan eventos.
No hagas grep del JSONL directamente para obtener el estado — usá `apx task list` o la API.

## `apx task add`

```bash
apx task add "<titulo>" \
  [--project <name|id|path>] \
  [--body <text>] \
  [--tag <name>]... \
  [--due <YYYY-MM-DD>] \
  [--agent <slug>] \
  [--source <label>]
```

`--tag` es repetible. Ejemplos:

```bash
apx task add "Revisar PR #42" --project myapp --agent reviewer --tag review
apx task add "Release notes"  --project myapp --tag release --tag docs --due 2026-06-01
apx task add "Llamar al cliente" --project myapp --due 2026-05-31 --tag urgent
```

<Aside type="caution">
  Siempre pasá `--project`. Sin él, la task cae en el primer proyecto registrado (o default id=0),
  que probablemente no es lo que querés.
</Aside>

## `apx task list`

```bash
apx task list \
  [--state open|done|dropped|all] \   # default: open
  [--tag <name>] \
  [--agent <slug>] \
  [--due-before <iso>] \
  [--limit <N>] \
  [--project <name|id|path>]
```

```bash
apx task list --project myapp                          # tareas abiertas
apx task list --project myapp --state all
apx task list --project myapp --state done
apx task list --project myapp --tag urgent
apx task list --project myapp --due-before 2026-06-01
apx task list --project myapp --agent reviewer --limit 10
```

## `apx task show`

```bash
apx task show <id> [--project <name|id|path>]
apx task show abc  [--project <name|id|path>]   # prefijo (≥ 3 chars, debe ser único)
```

Imprime la task completa como JSON, incluyendo todos los campos y el estado actual.

## Transiciones de estado

```bash
apx task done   <id> [--project P] [--by <name>]   # marcar completada
apx task drop   <id> [--project P]                  # archivar (ya no es necesaria)
apx task reopen <id> [--project P]                  # volver a abierta
```

`done` significa "completé este trabajo." `drop` significa "ya no es necesaria." Los reportes y
métricas los distinguen — usá el correcto.

## `apx task patch`

```bash
apx task patch <id> \
  [--title <text>] \
  [--body <text>] \
  [--due <date>] \
  [--agent <slug>] \
  [--tag <name>]... \
  [--project <name|id|path>]
```

`--tag` _reemplaza_ la lista de tags cuando se pasa; no pasar ningún `--tag` deja los tags sin
cambios.

```bash
apx task patch t_abc123 --project myapp --title "Nuevo título"
apx task patch t_abc123 --project myapp --tag review --tag urgent   # reemplaza tags
apx task patch t_abc123 --project myapp --due 2026-06-10
```

## Formato de ID y direccionamiento por prefijo

Los IDs de task tienen la forma `t_` + 6 caracteres base36 (entropía de 32 bits, ~4 mil millones
de keyspace). Podés direccionar una task por cualquier prefijo de ≥ 3 caracteres siempre que
identifique únicamente a una sola task. Si dos tasks comparten un prefijo, el comando devuelve
un error — usá un prefijo más largo.

```bash
apx task done t_abc123 --project myapp     # id completo
apx task done abc      --project myapp     # prefijo, resuelve si es único
```

## Campos de la task

| Campo | Cuándo | Notas |
|---|---|---|
| `title` | Requerido | Línea corta en modo imperativo. |
| `body` | Opcional | Notas más largas. Markdown aceptado. |
| `tags` | Opcional | Strings libres. Filtrables con `--tag`. |
| `due` | Opcional | Fecha ISO `YYYY-MM-DD`. Filtrable con `--due-before`. |
| `agent` | Opcional | Slug del agente responsable. |
| `source` | Auto / opcional | Origen: `cli`, `telegram`, `super-agent`, … |
| `state` | Derivado | `open` → `done` o `dropped`. Reabribles. |

## Cómo el super-agent alimenta las tasks

El super-agent tiene las herramientas `create_task` y `list_tasks` registradas en su conjunto
core de herramientas, disponibles en todos los canales. Cuando le decís "anotame que hay que
cerrar el bug de auth en myapp", el modelo llama a `create_task` con el proyecto correcto, título
y campos opcionales. Cuando preguntás "¿qué tengo pendiente en myapp?", llama a `list_tasks`.

Ejemplo de tool call que emite el modelo:

```json
{
  "name": "create_task",
  "arguments": {
    "project": "myapp",
    "title": "Cerrar bug de auth",
    "due": "2026-06-01",
    "tags": ["bug"]
  }
}
```

Si el proyecto es ambiguo (el usuario no dijo cuál), el modelo llama a `list_projects` primero y
pregunta — nunca asume. En un canal de Telegram anclado a un proyecto, el modelo usa ese proyecto
como contexto por defecto.

<Aside type="note">
  Las tasks no son un scheduler — no disparan notificaciones cuando vencen. Son una lista. Un
  kind de routine futuro (`task-due-notify`) está planeado pero todavía no está construido.
</Aside>
