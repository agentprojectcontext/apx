El sistema de memoria de APX es una capa de cuatro piezas que le da al super-agent contexto
durable y cross-channel sin ninguna curación manual. Corre en segundo plano en cada turno y
degrada de forma elegante — si una pieza falla, el resto sigue funcionando.

<Aside type="note">
  Esta página cubre el sistema de memoria automático, cross-channel. Para el archivo `memory.md`
  curado por agente que editás a mano, ver [Memoria (conceptos)](/apx/es/concepts/memory/).
</Aside>

## Las cuatro piezas

### Pieza 1 — Notas auto-escritas (`~/.apx/memory.md`)

Cuando el super-agent llama a su herramienta `remember`, el hecho se agrega a
`~/.apx/memory.md` bajo una entrada con fecha. El broker (Pieza 4) también lee este archivo
directamente, así que las notas manuales recientes siempre aparecen en el contexto.

```bash
# Desde una conversación: "remember que la dirección del servidor de staging cambió a 10.0.1.5"
# El agente llama a la herramienta `remember` — vos no corrés esto directamente.
```

El archivo se crea al arrancar el daemon si no existe. _No_ se commitea al repo — es estado
global local a tu máquina.

### Pieza 2 — Recuperación RAG (búsqueda vectorial)

Cada mensaje que pasa por cualquier canal se indexa de forma asíncrona en un vector store en
`~/.apx/memory.db` (SQLite via `better-sqlite3`). El indexer corre en segundo plano cada 60
segundos por defecto e indexa mensajes nuevos desde su última posición de cursor.

En el momento de consulta, el broker embedea el mensaje entrante y corre una búsqueda de
similitud coseno para recuperar los top-K chunks más relevantes de toda la historia cross-channel.

#### Provider de embeddings

El provider de embeddings se configura en `config.memory.embeddings` y se resuelve a través del
mismo patrón de registro de motores que TTS/STT. Providers soportados:

| Provider | Notas |
|---|---|
| `ollama` | Local, preferencia por defecto. Modelo: `nomic-embed-text`. |
| `openai` | Cloud. Modelo: `text-embedding-3-small` por defecto. |
| `gemini` | Cloud. Usa el endpoint de embeddings de Gemini. |
| `tf` | Fallback offline. Vector TF determinístico con feature-hashing. Siempre disponible. |

El modo de selección por defecto es `"auto"` (chain router): APX prueba `ollama → gemini →
openai → tf` en orden y usa el primero disponible. Podés anclar un provider único:

```json
{
  "memory": {
    "embeddings": {
      "provider": "openai",
      "openai": {
        "model": "text-embedding-3-small",
        "api_key": "sk-..."
      }
    }
  }
}
```

<Aside type="caution">
  Cambiar el provider de embeddings invalida el índice anterior — los vectores de modelos
  distintos no son comparables. Después de cambiar, ejecutá:

  ```bash
  apx memory reindex
  ```

  Esto limpia el store y re-embeda cada mensaje bajo el nuevo provider.
</Aside>

#### El fallback `tf`

Cuando ningún provider es alcanzable, APX cae silenciosamente en un vector de frecuencia de
términos con feature-hashing determinístico, sin dependencias (256-dim, L2-normalizado). La
calidad de recuperación es menor que un embedding neural real pero mantiene el sistema funcional
offline. La etiqueta `embedder` en cada vector almacenado garantiza que la similitud coseno solo
se calcule dentro del mismo espacio de embedder.

### Pieza 3 — Compactación progresiva

Las conversaciones largas hacen crecer la ventana de contexto. Cuando el chat de un canal acumula
más de 60 turnos conversacionales (configurable), los turnos más viejos más allá de los 40 más
recientes (configurable) se colapsan en un resumen denso escrito por un LLM ligero (registro
`type: "compact"` en el JSONL). Los turnos futuros prependen ese resumen como un turno de sistema
`[RESUMEN COMPACTADO]`, manteniendo el contexto acotado pero preservando decisiones, asignaciones
de tareas y resultados de herramientas.

La compactación corre _fuera del camino caliente de respuesta_ — el turno actual usa el compact
existente; el siguiente turno se beneficia del nuevo.

El modelo de compactación es configurable:

```json
{
  "memory": {
    "compact_model": "ollama:gemma4:31b-cloud",
    "compact_fallback_model": "",
    "compact_threshold": 60,
    "keep_recent": 40
  }
}
```

Si tanto `compact_model` como `compact_fallback_model` no están disponibles, la compactación se
omite silenciosamente — los turnos crudos se preservan y ninguna respuesta se bloquea.

### Pieza 4 — Memory broker

Antes de cada turno del super-agent (en llamadas no tool-free), el broker ensambla un bloque
`[MEMORIA RELEVANTE]` que se inyecta en el system prompt. El broker:

1. Lee las últimas 10 entradas de `~/.apx/memory.md` (sincrónico, siempre rápido).
2. Corre una consulta RAG contra un presupuesto de 800 ms — si Ollama está lento, el bloque
   igual se devuelve con lo que `memory.md` aportó.
3. Deduplica hits por texto normalizado.
4. Formatea una lista con viñetas con fecha, canal y un extracto de 160 chars por entrada.

El bloque se omite completamente cuando no hay nada útil para mostrar (`memory.md` vacío y sin
hits RAG).

#### Bloque de hilos activos

Además del bloque de memoria, en canales interactivos (no `routine`) el broker agrega un bloque
separado "# Hilos activos en otros canales" — el turno más reciente de cada otro canal dentro de
una ventana de tiempo configurable (default: 6 horas, hasta 3 viñetas). Esto ayuda al agente a
notar referencias del tipo "lo de antes en Telegram" sin necesitar un match semántico.

## Referencia de configuración

Todas las claves están bajo `config.memory`:

| Clave | Default | Descripción |
|---|---|---|
| `enabled` | `true` | Setear a `false` para deshabilitar todo el subsistema RAG. |
| `index_interval_s` | `60` | Frecuencia del indexer en background (segundos). |
| `broker_budget_ms` | `800` | Tiempo máximo que el broker espera el RAG antes de devolver. |
| `rag_top_k` | `5` | Número de hits RAG inyectados por turno. |
| `compact_threshold` | `60` | Turnos antes de que se dispare la compactación. |
| `keep_recent` | `40` | Turnos que quedan verbatim después de la compactación. |
| `compact_model` | `ollama:gemma4:31b-cloud` | Modelo primario para el resumen. |
| `compact_fallback_model` | _(super_agent.model)_ | Fallback si el primario está caído. |
| `embeddings.provider` | `"auto"` | `auto`, `ollama`, `openai`, `gemini`, o `tf`. |
| `active_threads.enabled` | `true` | Mostrar turnos recientes de otros canales. |
| `active_threads.window_hours` | `6` | Hasta cuánto tiempo atrás buscar en otros canales. |
| `active_threads.max_lines` | `3` | Máximo de viñetas en el bloque de hilos activos. |

## Diferencia con el memory.md por agente

| | `memory.md` por agente | Sistema de memoria cross-channel |
|---|---|---|
| **Ubicación** | `.apc/agents/<slug>/memory.md` | `~/.apx/memory.md` + `~/.apx/memory.db` |
| **¿Se commitea?** | Sí | Nunca |
| **Alcance** | Un agente | Todos los canales, todos los agentes |
| **Lo escribe** | Vos, a mano | La herramienta `remember` (automático) |
| **Recuperación** | Inyección completa en cada llamada | RAG (top-K, con score) + últimas 10 notas |
| **Compactación** | No aplica | Progresiva (Pieza 3) |

Usá el archivo por agente para hechos estables y curados sobre el rol del agente. El sistema
cross-channel maneja el contexto dinámico, turno a turno, que abarca superficies.
