Una **sesión** es el registro de una invocación de agente o un hilo de trabajo. Las sesiones
nativas de APX son archivos Markdown guardados localmente — nunca se commitean. APX también entiende
sesiones creadas por Claude Code y Codex, dándote una interfaz unificada para encontrar y retomar
el trabajo sin importar qué engine lo inició.

## Sesiones vs. conversaciones

| Concepto | Qué es | Dónde vive |
|---|---|---|
| Sesión | Registro de alto nivel — título, estado, resultado, timestamps | `~/.apx/projects/<id>/agents/<slug>/sessions/` |
| Conversación | El thread crudo de mensajes del LLM | `~/.apx/projects/<id>/agents/<slug>/conversations/` |
| Sesión externa | Archivo de sesión de Claude o Codex | `~/.claude/` o `~/.codex/` |

Las sesiones son el artefacto durable al que volvés; las conversaciones son el log de turnos
que lee el engine.

## Cobertura de engines

Los comandos cross-engine de APX escanean estas ubicaciones:

| Engine | Dónde mira APX |
|---|---|
| `apx` | `~/.apx/projects/<apx_id>/agents/<slug>/sessions/*.md` |
| `claude` | `~/.claude/projects/<encoded-cwd>/<id>.jsonl` |
| `codex` | `~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl` |

Los engines que no están instalados se omiten silenciosamente.

## Encontrar sesiones

Empezá con `apx session find` cuando recordás el tema pero no el id:

```bash
apx session find "mejorar interfaz web"        # búsqueda por título, cross-engine, más reciente primero
apx session find "sidebar" --deep              # también busca dentro del contenido del transcript
apx session find "deploy" --engine codex       # restringir a un solo engine
apx session find "api" --dir /path/to/repo     # acotar a un directorio
apx session find "login" --limit 5 --json      # salida legible por máquina
```

La salida incluye un bloque de comandos `summary`/`ask`/`resume` listos para ejecutar,
pre-cargados con el id del mejor resultado — podés copiar y pegar directamente.

## Listar sesiones

```bash
apx sessions list                              # todos los engines, todos los proyectos conocidos
apx sessions list --engine claude              # sólo sesiones de Claude
apx sessions list --engine codex --limit 10
apx sessions list --project my-app            # todos los engines, un proyecto registrado
apx sessions list --dir /path/to/repo         # todos los engines, un directorio
```

## Comandos nativos de sesión APX

Estos gestionan los archivos `.md` de sesión que APX escribe bajo `~/.apx/`:

```bash
apx session new <slug> --title "Investigar bug X"     # crear un archivo de sesión
apx session list                                       # todos los agentes en el proyecto actual
apx session list <slug>                               # sesiones de un agente
apx session get <id>                                  # metadata
apx session get <id> --body                           # cuerpo markdown completo
apx session update <id> --status "in progress"
apx session close <id> --result "Resuelto en PR #42"
apx session check                                     # sale con error 1 si hay una sesión activa
apx session close-stale                               # auto-cerrar sesiones de más de 1 hora
apx session compact <slug>                            # resumir una conversación a disco
```

## Resume cross-engine

```bash
apx session resume <id>                    # auto-detectar engine, mostrar metadata
apx session resume <id> --tail 32k         # imprimir los últimos 32 KB del transcript (sin daemon)
apx session resume <id> --full             # volcar el transcript completo
apx session resume <id> --summary          # resumen LLM vía super-agente (requiere daemon)
apx session resume <id> --continue         # lanzar la CLI nativa (claude --resume / codex resume)
apx session resume <id> --into apx:slug    # crear una sesión APX nueva desde el resumen
apx session resume <id> --engine claude    # saltar auto-detección, usar un engine específico
```

Si el mismo id aparece en múltiples engines, APX muestra todos los matches y te pide que
re-ejecutes con `--engine <id>` para desambiguar.

## Leer el contenido de una sesión

```bash
# Sesiones locales de APX
apx session get <id>                       # metadata
apx session get <id> --body               # markdown completo

# Cualquier engine
apx session get <id> --engine claude --full    # JSONL completo de Claude
apx session get <id> --engine codex --tail 16k
apx session get <id> --any --full             # buscar en todos los engines, error si hay colisión
```

## Resumir y preguntar

Estos dos comandos requieren el daemon y `super_agent.enabled: true` en `~/.apx/config.json`:

```bash
apx session summary <id>                           # resumen LLM en 4 bullets
apx session summary <id> --max-chunks 8            # limitar el costo en transcripts grandes

apx session ask <id> "¿qué cambios al sidebar?"   # Q&A sobre el transcript
apx session ask <id> "what files changed?" --max-chunks 30
```

Los transcripts grandes se procesan con map-reduce: cada chunk de ~48 KB se analiza en busca de
notas relevantes, y un pase final sintetiza la respuesta. El ruido binario (payloads de imágenes
en base64) se elimina antes del chunking.

<Aside type="tip">
  Usá `apx session summary` para orientarte y luego `apx session ask` para preguntas específicas.
  En rollouts de Codex muy grandes, subí `--max-chunks` si la salida avisa sobre truncamiento.
</Aside>

## Requisitos del daemon

| Comando | ¿Requiere daemon? |
|---|---|
| `apx session find` | No |
| `apx sessions list` | No |
| `apx session get` (cualquier modo) | No |
| `apx session resume` (metadata, tail, full, continue) | No |
| `apx session resume --summary` / `apx session summary` | Sí |
| `apx session ask` | Sí |
| `apx session resume --into apx:slug` (sólo la parte del resumen) | Sí |

El daemon arranca automáticamente cuando se necesita — no hace falta lanzarlo a mano.
