El daemon APX (`apx-daemon`) es la **capa host** del sistema: un servidor HTTP local de larga
duración en `127.0.0.1:7430` que administra todo lo que necesita persistir entre comandos: estado
de proyectos, ciclo de vida de plugins, el scheduler, archivos de conversación y el endpoint del
super-agent.

Cada superficie — el CLI `apx`, el TUI, el panel web de administración, el bridge MCP, la ventana
Desktop — se comunica con el daemon por HTTP. Ver [Arquitectura](/apx/es/start/architecture/) para
el panorama completo.

## Auto-inicio

Nunca necesitás iniciar el daemon manualmente. La primera llamada a `apx` que lo necesita lo
lanza en segundo plano y espera un momento a que esté saludable. Las llamadas siguientes reutilizan
el proceso en ejecución.

El daemon es un **singleton**: un archivo PID en `~/.apx/daemon.pid` previene arranques duplicados.
Si el archivo existe pero el proceso está muerto, el próximo inicio lo limpia automáticamente.

<Aside type="tip">
  Ejecutá `apx status` en cualquier momento para ver si el daemon está corriendo, qué plugins están
  activos y cuántos proyectos están cargados.
</Aside>

## Gestionar el daemon

```bash
apx daemon start      # iniciar el daemon explícitamente
apx daemon stop       # enviar SIGTERM y esperar
apx daemon reload     # volver a leer ~/.apx/config.json — NO reinicia el proceso
apx daemon status     # mostrar estado: puerto, pid, uptime, plugins, proyectos
apx daemon logs       # tail del log stdout del daemon
apx daemon logs --tail 200
```

`apx daemon reload` es la opción ligera: recarga cambios de config (canales Telegram, claves de
modelos, modo de permisos) sin interrumpir las requests en vuelo. Usá `apx daemon restart`
(un stop + start) cuando cambiás código fuente o prompts.

## Logs

APX escribe en dos lugares:

| Ruta | Qué contiene | Comando |
|------|------|---------|
| `~/.apx/daemon.log` | Redireccionamiento stdout del proceso daemon | `apx daemon logs --tail N` |
| `~/.apx/logs/apx.log` | Log estructurado unificado — todos los módulos escriben aquí | `apx log` |

**`apx log` es la forma recomendada de seguir el sistema:**

```bash
apx log              # muestra las últimas 200 líneas
apx log -f           # sigue el log (estilo tail -f)
apx log --tail 500
apx log --errors     # solo líneas de nivel ERROR
```

El formato del log unificado es:

```text
[2026-05-30 14:22:01.123] [INFO ] [telegram] plugin telegram initialized
[2026-05-30 14:22:01.456] [INFO ] [daemon  ] apx-daemon 1.22.2 listening on http://127.0.0.1:7430
```

Los secretos (tokens, API keys) se redactan automáticamente antes de escribir.

## Ciclo de vida de los plugins

Al arrancar, el daemon descubre dos plugins integrados:

| Plugin | Qué hace |
|--------|----------|
| `telegram` | Gateway de bot Telegram con long-poll, enrutamiento de mensajes entrantes, roster de contactos |
| `desktop` | Bridge WebSocket para la ventana flotante Electron |

El `PluginManager` llama al `init()` de cada plugin una vez, y luego a `start()` después de que
el servidor HTTP esté escuchando. En `SIGTERM` / `SIGINT` llama a `stop()` en cada plugin antes
de cerrar el servidor.

Los plugins también pueden instalar sus propias rutas Express (hook `routes(app)`). El daemon las
registra inmediatamente después de las rutas de la API principal, por lo que las rutas de los
plugins están disponibles en la misma URL base.

Podés inspeccionar el estado de los plugins en cualquier momento:

```bash
apx plugins list            # todos los plugins cargados + running/stopped
apx plugins status telegram # estado detallado de un plugin
```

## Ticks del scheduler

El daemon corre un `RoutineScheduler` que hace tick cada 5 segundos y verifica qué rutinas tienen
que ejecutarse. Cuando una rutina se dispara, el scheduler:

1. Lee la definición de la rutina desde `.apc/routines.json`.
2. Resuelve el handler del kind (`heartbeat`, `exec_agent`, `super_agent`, `telegram`, `shell`).
3. Ejecuta los `pre_commands` configurados, evalúa la condición `skip_prompt_on`, corre la acción
   principal y luego corre los `post_commands`.
4. Escribe los timestamps actualizados `last_run` / `next_run` de vuelta en el store de rutinas.

Ver [Routines](/apx/es/capabilities/routines/) para la spec completa y los flags de `apx routine add`.

## Archivos de conversación

Cada hilo de conversación de agente se guarda como un archivo Markdown en disco — no se necesita
base de datos para los datos principales, solo para el índice regenerable. El layout:

<FileTree>
- ~/.apx/
  - config.json
  - identity.json
  - daemon.pid
  - daemon.token
  - daemon.log
  - logs/
    - apx.log
    - errors.jsonl
  - messages/
    - telegram/
      - YYYY-MM-DD.jsonl
  - projects/
    - \<project-id\>/
      - project.db  ← índice SQLite regenerable
      - messages/
      - agents/
        - \<slug\>/
          - sessions/   ← un .md por invocación de runtime
          - conversations/  ← hilos de conversación LLM
        - default/
          - sessions/
</FileTree>

<Aside type="note">
  Nada de lo que está bajo `~/.apx/` se commitea a git. Las definiciones de proyectos y la memoria
  curada viven en el repositorio bajo `.apc/`. Ver [Arquitectura](/apx/es/start/architecture/) para
  la justificación del split de almacenamiento.
</Aside>

## Configuración

El daemon lee `~/.apx/config.json` al arrancar. Campos clave:

```json
{
  "port": 7430,
  "host": "127.0.0.1",
  "log_level": "info",
  "super_agent": {
    "enabled": true,
    "model": "claude-3-5-sonnet",
    "permission_mode": "automatico"
  },
  "telegram": { "enabled": true, "channels": [] },
  "engines": {
    "anthropic": { "api_key": "sk-ant-..." }
  }
}
```

Después de editar la config, ejecutá `apx daemon reload` (para cambios solo de config) o
`apx daemon restart` (para cambios de código o prompts). Los comandos `apx config set` y
`apx model key` escriben este archivo y llaman al endpoint de reload automáticamente.
