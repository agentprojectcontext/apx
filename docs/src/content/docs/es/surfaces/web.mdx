El panel web de administración es la interfaz de APX para el navegador. Corre
completamente en tu browser, se comunica con el daemon a través de su HTTP API
y no requiere ningún servidor externo. Cualquier navegador en la misma máquina
— o en la misma LAN después de vincular — puede usarlo.

## Abrir en la misma máquina

El daemon sirve el panel compilado en `http://127.0.0.1:7430/`. Arrancá el
daemon y abrí esa URL:

```bash
apx status          # arranca el daemon si no está corriendo
# luego abrí http://127.0.0.1:7430 en el navegador
```

En el primer acceso el panel obtiene el token de autenticación automáticamente
desde `/admin/web-token` (endpoint exclusivo para loopback). No tenés que copiar
ningún token manualmente cuando lo abrís desde la misma máquina.

<Aside type="note">
  Si el panel muestra "The admin panel hasn't been built on this install",
  ejecutá `node scripts/build-web.js` (o `npm run build:web`) desde la raíz del
  repo para compilar el bundle en `src/interfaces/web/dist/`.
</Aside>

## Abrir desde otro dispositivo (vinculación LAN)

Para abrir el panel en tu teléfono o en otra computadora de la misma LAN, usá
`apx pair web`. El comando imprime un código QR y una URL directa:

```bash
apx pair web
# Imprime:
#   http://192.168.1.42:7430/#token=<token>
#   [código QR para escanear con la cámara del teléfono]
```

Escaneá el QR con la cámara del teléfono (sin app) o compartí la URL. El
fragmento `#token=…` lleva el token — el panel lo usa automáticamente.

<Screenshot
  surface="terminal"
  caption="apx pair web — código QR y link LAN impresos en la terminal"
  hint="Ejecutá: apx pair web" />

Para ver los clientes vinculados o revocar uno:

```bash
apx pair list
apx pair revoke <id>
```

<Aside type="caution">
  El daemon se vincula a `127.0.0.1` por defecto. Para acceso LAN tenés que
  habilitarlo explícitamente: `apx config set remote.bind 0.0.0.0` y después
  reiniciá el daemon. Es un paso deliberado para que no expongas el daemon en
  una red compartida sin querer.
</Aside>

## Compilar el bundle web

El panel es una app Vite + React + TypeScript en `src/interfaces/web/`. El
resultado compilado queda en `src/interfaces/web/dist/` y lo sirve el daemon.

```bash
# Compilar una vez (desde la raíz del repo)
node scripts/build-web.js

# O compilar manualmente
cd src/interfaces/web
pnpm install
pnpm build

# Desarrollo con hot reload (proxea las llamadas API a :7430)
pnpm dev
```

El script `build:web` y el hook `prepack` llaman a `node scripts/build-web.js`,
así que un `npm install -g .` global siempre genera un bundle actualizado.
Usá `APX_SKIP_WEB_BUILD=1` para saltear la compilación en flujos de desarrollo
que no tocan la UI.

## Modelo de navegación

El panel tiene dos niveles de navegación:

- **Base** (`/p/0/…`) — el espacio global del daemon: workspaces, modelos,
  sesiones, logs y configuración global. El chat en Base habla con el
  super-agente.
- **Proyecto** (`/p/:pid/…`) — un workspace de proyecto: agentes, chat,
  tareas, rutinas, MCPs, memorias y configuración.

El rail izquierdo muestra todos los proyectos registrados más accesos directos
a módulos (Voces, Escritorio, Deck, Code). Abajo del rail hay un botón **Roby**
que abre un sheet flotante de chat conectado al super-agente.

## Módulos del rail

### Proyectos

La pestaña Proyectos (Base → Workspaces) lista todos los proyectos registrados
como cards. Hacé click en una card para entrar al proyecto. El botón "Nuevo
proyecto" abre un dialog inline (`?action=add-project`).

<Screenshot
  surface="web"
  caption="Pestaña Workspaces — cards de proyectos con nombre, tipo y ruta"
  hint="Navegá a /p/0/workspaces" />

### Agentes

Cada proyecto tiene una pestaña Agentes. Podés crear, editar y eliminar
agentes; ver y editar el `memory.md` de cada agente; y asignar skills. En el
espacio Base la pestaña de Agentes muestra la configuración del super-agente
(solo lectura).

<Screenshot
  surface="web"
  caption="Pestaña Agentes — lista de agentes con contadores de memoria y skills"
  hint="Navegá a /p/:pid/agents" />

### Voces

El módulo Voces (`/m/voice`) configura TTS y STT globalmente. Muestra el estado
en vivo de cada motor TTS (Piper, ElevenLabs, OpenAI, Gemini, mock), te permite
elegir un motor por defecto o modo cadena-con-fallback, reordenar la cadena,
configurar credenciales por motor y hacer un test de reproducción en vivo. La
sección STT configura el modelo de Whisper y el idioma para la entrada de
micrófono.

Mirá [Voice](/apx/es/surfaces/voice/) para la referencia completa de la CLI.

<Screenshot
  surface="web"
  caption="Módulo Voces — lista de proveedores TTS con badges de estado y selector de defecto"

### Escritorio

El módulo Escritorio (`/m/desktop`) muestra si la ventana flotante Electron está
corriendo (clientes WebSocket conectados), te deja editar el atajo global y
activar o desactivar que el daemon responda mensajes del escritorio. No podés
iniciar ni detener el proceso Electron desde el panel web — usá
`apx desktop start / stop`.

Mirá [Desktop](/apx/es/surfaces/desktop/) para la referencia completa de la CLI.

### Deck

El módulo Deck (`/m/deck`) muestra el payload de `GET /deck/manifest`: salud
del daemon, proyecto activo, plugins registrados y la grilla de widgets y
escritorios. Podés habilitar o deshabilitar widgets individuales desde esta
vista.

Mirá [Deck](/apx/es/surfaces/deck/) para la referencia de la CLI de vinculación.

<Screenshot
  surface="web"
  caption="Módulo Deck — vista del manifest con grilla de widgets y card de contexto APX"
  hint="Navegá a /m/deck" />

### Code

El módulo Code (`/m/code`) es la superficie de administración del asistente de
código APX (`apx code`). Te permite elegir un proyecto e inspeccionar el trail
de herramientas de sesiones recientes.

### Configuración

La configuración global vive en **Settings** (`/settings/*`). Subsecciones:

| Ruta | Qué edita |
|---|---|
| `/settings/identity` | Nombre del asistente, persona |
| `/settings/appearance` | Tema (claro/oscuro) |
| `/settings/super-agent` | Modelo del super-agente, system prompt, canales |
| `/settings/engines` | Proveedores de modelos (agregar, editar, activar) |
| `/settings/telegram` | Token del bot de Telegram y canales |
| `/settings/devices` | Lista de dispositivos vinculados |
| `/settings/advanced` | Editor raw de `~/.apx/config.json` |

## Arquitectura

El panel es un cliente fino: no tiene estado local más allá de la sesión
actual. Cada lectura es un GET contra la HTTP API del daemon; cada escritura
es un PATCH o POST. El daemon es la única fuente de verdad.

```text
Browser  →  GET/POST  →  http://127.0.0.1:7430/<ruta>
                              ↓
                         APX daemon (host/daemon/)
                              ↓
                         core/ (lógica, memoria, agentes…)
```

El bundle web lo sirve `src/host/daemon/api/web.js`. Las rutas de API siempre
tienen prioridad sobre el catch-all de la SPA, así que una request a
`/projects/1/agents` llega al endpoint real, no a `index.html`.
